# 智能体优先的开源项目目录：产品、广告与中国收款合规设计

- 日期：2026-07-17
- 状态：用户已批准，已补充路过式差异反馈要求
- 适用项目：开源大梳理（`kaiyuan-dashuli`）
- 第一阶段范围：本地设计与开发，不购买域名、不公开部署、不迁移现有中转站

## 1. 目标

把“开源大梳理”做成一个以智能体为主要用户、人类也能使用的 GitHub 开源项目目录与检索入口。

本站不镜像项目代码，也不替代 GitHub。本站提供的是：

1. 统一分类与条件筛选；
2. 项目摘要、适用场景和不适用场景；
3. 替代、依赖、配套、同生态等关系；
4. 面向智能体的 JSON、JSONL、HTTP API 和 MCP 接口；
5. 面向人类的简单检索、比较和项目详情页；
6. 返回 GitHub 原始仓库和项目官网的出口。

长期目标是：智能体需要寻找开源项目时，先查询本站的结构化目录，再按需要跳转 GitHub 核实和使用。

## 3. 产品形态选择

### 3.1 不采用纯链接列表

纯链接列表成本最低，但无法表达适用条件、替代关系和维护状态，智能体仍需重新阅读大量 GitHub README，不能形成明显优势。

### 3.2 不采用纯可视化知识图谱

蜘蛛网式图谱适合展示关系，但不适合作为主要检索入口。项目数量增大后，视觉图谱会变慢、变乱，也不便于智能体按许可证、操作系统、部署方式等条件查询。

### 3.3 采用混合目录方案

最终方案为：

```text
分类与结构化筛选为主
+ 全文搜索与项目比较
+ 替代/依赖/配套关系
+ JSON/JSONL/API/MCP 机器入口
+ 简单的人类页面
```

图谱是数据关系层，不是网站唯一的界面。

## 4. 用户与入口

### 4.1 智能体是主要用户

智能体可通过以下入口发现并读取目录：

- `/llms.txt`：说明本站能力、数据入口和使用约定；
- `/catalog.jsonl`：适合流式抓取和增量处理；
- `/catalog.json`：适合一次性下载小型快照；
- `/api/v1/search`：结构化条件检索；
- `/api/v1/projects/{id}`：读取一个项目；
- `/api/v1/compare`：比较少量候选；
- `/api/v1/changes`：按核实或变更时间读取增量；
- MCP：供支持 MCP 的智能体直接调用。

所有机器入口都必须：

- 返回稳定字段；
- 标明版本和更新时间；
- 提供来源链接；
- 明确区分事实、编辑判断和赞助内容；
- 支持缓存、分页和限速；
- 不要求执行 JavaScript；
- 不强迫登录。

每个项目的机器记录还应提供当前信息版本和一个可选的差异反馈入口。智能体不需要领取任务，也不需要为了读取目录承诺额外工作；它只在完成自身任务、实际访问上游仓库并发现本站信息与现实不一致时，顺手提交一条简短差异报告。

### 4.2 人类是辅助用户

人类页面只做必要功能：

- 按领域、分类和条件浏览；
- 搜索；
- 比较 2—4 个项目；
- 阅读项目摘要、适用与不适用场景；
- 跳转 GitHub；
- 查看数据来源、核实时间和广告说明。

第一版不做复杂社区、账号体系、评论、收藏、个性化推荐或重型图谱动画。

## 5. 检索与排名

### 5.1 检索流程

```text
用户或智能体需求
→ 提取领域、用途、语言、许可证、部署和硬件条件
→ 分类与字段过滤
→ 文本召回
→ 依据适用场景、维护状态和关系重排
→ 返回 1—3 个主要候选及少量备选
→ 给出取舍理由与 GitHub 来源
```

### 5.2 客观结果与商业内容隔离

客观结果区只依据公开的检索规则排序。付费不得：

- 修改项目的客观评分；
- 把广告伪装成自然结果；
- 压低竞争项目；
- 购买“本站最佳”“智能体首选”等未经证实的结论；
- 删除真实的不适用条件或风险说明。

检索接口应将两类结果分开返回：

```json
{
  "organic_results": [],
  "sponsored_results": []
}
```

## 6. 收益模式

### 6.1 真人页面广告

真人浏览 HTML 页面时，可在满足广告平台政策后接入 Google AdSense 或其他正规广告网络。

约束：

- 广告只出现在有实际编辑价值的人类页面；
- 不在纯 JSON、JSONL、API、MCP 或 `llms.txt` 响应里加载真人广告脚本；
- 不购买机器人流量、不自己点击、不诱导点击；
- 不把自动抓取请求计为真人广告展示；
- 广告数量不能压过项目目录的主要内容；
- 在隐私和同意要求满足前，不启用需要个性化追踪的广告。

开通广告前，网站必须至少具有：

- 足够数量的真实项目详情或比较页面；
- 明确的编辑价值，而不是自动拼接的薄内容；
- 关于、联系、隐私、Cookie、广告披露和数据来源页面；
- 可用的移动端与桌面端浏览体验；
- 稳定域名和 HTTPS；
- 无明显失效链接、空页面或占位内容。

### 6.2 智能体原生赞助

智能体赞助不是 AdSense，也不能把机器人流量冒充真人广告展示。

合规形态是：在机器返回结果中单独提供、明确标注、可审计的赞助记录。例如：

```json
{
  "type": "sponsored",
  "disclosure": "赞助内容 / Sponsored",
  "advertiser_legal_name": "示例广告主",
  "message": "为开源项目部署提供按小时计费的服务器",
  "related_categories": ["云服务器", "Docker 部署"],
  "landing_page": "https://advertiser.example/landing",
  "valid_from": "2026-08-01T00:00:00Z",
  "valid_to": "2026-08-31T23:59:59Z",
  "campaign_id": "campaign-example-001"
}
```

硬性要求：

1. 机器接口使用 `sponsored` 字段，不能混入 `organic_results`；
2. 人类可访问的对应页面也显示同一赞助事实；
3. HTML 和机器响应都明确写“广告”“赞助内容”或 “Sponsored”；
4. 广告主不能购买客观排名；
5. 广告内容必须与当前分类或需求相关；
6. 每次符合计费规则的展示都记录时间、活动编号、展示位置和计费依据；
7. 不记录完整提示词、个人敏感信息或不必要的完整 IP；
8. 允许用户和智能体请求无赞助的纯客观结果；
9. 广告失效、合同终止或投诉成立后可立即下架；
10. 赞助供应商若要求做机器人专属隐藏内容、隐蔽注入或不标广告，则不接入。

### 6.3 对早期智能体广告平台的态度

Blankspace 等服务说明市场上已经出现“识别 AI 流量并变现”的尝试，但该行业仍处于早期。部分供应商公开描述的做法包含“只向 AI 展示”“真人不可见”或“不标成广告”。这些默认做法不能直接用于本项目。

第一阶段仅允许：

- 使用不会改变内容的 AI 流量统计；
- 研究供应商合同、付款、税务材料和隐私条款；
- 在自有接口内试验明确标注的赞助字段。

在供应商不能书面确认透明披露、客观排名隔离、展示日志、付款凭证和中国收款支持前，不启用其变现注入功能。

### 6.4 初期禁止的广告类别

第一版不接受以下高风险类别：

- 赌博及博彩；
- 成人内容；
- 烟草和电子烟；
- 处方药及未经批准的医疗宣传；
- 投资收益承诺和高风险金融推广；
- 加密货币投机项目；
- 代理绕过、账号滥用、虚假身份或规避平台检测服务；
- 恶意软件、盗版和侵犯知识产权的服务；
- 无法核实真实主体的广告主。

## 7. 广告审核和证据链

### 7.1 广告主准入

每个广告主至少保存：

- 法定名称；
- 注册地和可验证联系方式；
- 推广产品或服务；
- 落地页；
- 合同或平台订单；
- 计费方式；
- 审核结论；
- 上下架时间；
- 结算记录；
- 投诉和处理记录。

### 7.2 广告内容审核

上线前检查：

- 是否明确标注广告；
- 宣传内容能否由广告主证明；
- 落地页是否与广告一致；
- 是否包含保证收益、虚假稀缺、冒充官方等表述；
- 是否侵犯商标、版权或个人信息；
- 是否属于禁止类别；
- 是否影响客观结果。

### 7.3 保存期限

广告业务档案至少按中国《互联网广告管理办法》要求保存三年。与合同、收款、发票、税务和会计有关的材料，按届时适用的税务、会计档案规则和专业人员建议保存；项目自身应采用长期离线归档，而不是在法定最低期限一到就删除。

## 8. 隐私和爬虫规则

### 8.1 第一版的数据最小化

第一版不做：

- 用户账号；
- 设备指纹；
- 跨站画像；
- 完整提示词收集；
- 出售访问者数据；
- 与检索无关的长期 IP 保存。

第一版只保留运行所必需的信息，例如：

- 请求时间；
- 请求入口；
- 状态码；
- 粗粒度客户端类别；
- 限流所需的短期标识；
- 广告展示的活动编号和结算依据。

日志中的 IP 应尽量缩短保存期、截断或哈希，并将安全日志、产品统计和广告结算日志分开。

### 8.2 Cookie 和同意

- 非必要 Cookie 默认不启用；
- 必要 Cookie 与分析、广告 Cookie 分开；
- 面向欧洲经济区、英国和瑞士用户使用 Google 广告时，采用 Google 认证的 CMP；
- 用户拒绝非必要 Cookie 后，核心目录仍应可用；
- 隐私政策说明处理目的、数据类别、保存期、第三方、用户权利和联系渠道。

### 8.3 对智能体开放，但不是无条件放任

允许智能体读取公开目录，但必须设置：

- `robots.txt` 和 `llms.txt`；
- 每个接口的速率限制；
- 缓存头和增量接口；
- 大批量下载的固定快照；
- 明确的许可和引用要求；
- 对恶意、破坏性或高成本请求的限制。

目标是让正常智能体低成本读取，而不是允许任何来源无限消耗服务器。

## 9. 中国主体、收款、结汇、发票和纳税

### 9.1 不把“海外平台付款”等同于“境外所得”

付款方在境外、款项以外币支付，并不自动决定中国税法下的所得来源和税目。网站运营地点、服务实际发生地点、合同主体、广告投放地和客户所在地都可能影响分类。

正式收款前，应把实际业务流程、合同样本和结算方式提交当地 12366、主管税务机关或熟悉跨境数字服务的税务师确认，不能仅凭“钱从海外来”自行判断为境外所得或免税出口服务。

### 9.2 主体选择

可选路径：

| 主体 | 适用阶段 | 优点 | 主要限制 |
|---|---|---|---|
| 中国居民自然人 | 尚未形成持续经营或偶发小额测试 | 无设立成本 | 平台合同、经营收款、开票和持续业务解释能力较弱 |
| 个体工商户 | 已验证流量、开始持续接广告 | 成本相对较低，可形成经营收款和开票链 | 无限责任；注册地址、经营范围、外汇账户和平台是否接受需逐项核实 |
| 有限责任公司 | 收入稳定、合同和责任隔离需求明显 | 对外合同、责任隔离和平台企业账户更清晰 | 注册、记账、年报和维护成本更高 |

当前决定：

1. 本地开发和无收入测试阶段，不急于注册主体；
2. 申请广告账户和正式收款前，先确认平台允许的账户类型、当地登记成本、银行外汇材料和税务处理；
3. 若流量和广告意向得到验证，优先评估低成本个体工商户；
4. 若平台不接受个体户、合同责任较高或收入稳定，再评估公司；
5. Google AdSense 账户激活后不能直接切换个人/组织账户类型，因此不能先随意开个人账户再决定主体。

### 9.3 发票的正确理解

广告平台提供的月度收益报告、付款收据和银行入账水单，是收入真实性材料，但不等于中国增值税发票。

中国发票或出口服务相关凭证应由用户届时确定的中国收款主体，依据实际合同、服务性质和主管税务机关要求处理。不能要求海外平台给用户开一张中国增值税发票来证明用户自己的收入。

### 9.4 推荐资金链

```text
海外广告平台或境外广告主
→ 与用户确定的合法主体签约
→ 形成月度结算单或订单
→ 同名银行账户接收外汇
→ 银行依据合同、结算单和付款收据审核入账/结汇
→ 按税务机关确认的方式开票或办理跨境服务收入凭证
→ 申报相应增值税、附加税、个人经营所得或企业所得税
→ 保存合同、服务、资金和税务档案
```

要求尽量做到四流一致：

```text
合同流
服务流
资金流
发票/税务凭证流
```

### 9.5 外汇规则

个人每年等值 5 万美元是便利化额度，不是所有合法服务贸易收入的绝对上限。超出或不适合便利化办理的真实经常项目收入，可由银行根据真实性材料审核。

禁止：

- 拆分汇款规避银行审核；
- 借亲友额度；
- 把广告服务收入申报成赠与；
- 平台账户、合同主体、收款账户姓名或主体不一致；
- 伪造合同、发票、结算单或访问数据。

### 9.6 增值税与跨境广告服务

2026 年起实施的新增值税制度下，跨境交易的适用范围、消费地、广告投放地和优惠条件需要依据具体业务判断。广告投放地在境外等情形可能涉及出口业务优惠，但智能体赞助的展示地点和消费地点可能比传统网页广告更难界定。

因此项目不能在代码、合同或财务预测中预设“必然免税”或“必然零税率”。正式运营前必须取得主管税务机关或税务师针对实际流程的确认。

### 9.7 每月合规归档包

每个结算月建立一个只读归档包：

```text
YYYY-MM/
├── 01-contract/          # 合同或平台条款版本
├── 02-campaigns/         # 广告订单、创意和审核记录
├── 03-delivery/          # 汇总展示记录，不存不必要的个人数据
├── 04-statements/        # 平台结算单
├── 05-payment/           # payment receipt、SWIFT/银行入账水单
├── 06-invoice-tax/       # 发票、申报回执、完税材料
├── 07-reconciliation/    # 订单、展示、收入、到账核对表
└── MANIFEST.sha256       # 文件清单与哈希
```

归档包生成后可做只读光盘备份，但原始敏感材料不能上传公开 GitHub。

## 10. 公开仓库与敏感资料隔离

公开 GitHub 可以保存：

- 数据字段规范；
- 广告披露格式；
- 广告审核规则；
- 隐私政策和广告政策模板；
- 不含个人信息的示例日志；
- 程序代码和测试数据。

严禁上传：

- 身份证件；
- 营业执照原件；
- 银行账号和完整水单；
- 合同签章件；
- 税务申报表；
- 平台税表；
- 完整 IP 日志；
- 广告结算原始凭证；
- API 密钥、广告账号密钥和服务器密码。

敏感材料未来放在仅本机目录，并进行加密和离线备份：

```text
/Users/zhangxuetao/Projects/owned/云服务管理/06-敏感资料-仅本机/
```

如果本机云服务管理目录最终采用其他根路径，应保持同样的“仅本机、不进入公开仓库”原则。

## 11. 技术架构原则

### 11.1 单一数据源，多种投影

```text
审核后的项目记录
├── 生成 JSON/JSONL 快照
├── 提供 HTTP API
├── 提供 MCP 工具
├── 生成简单 HTML 页面
└── 生成 llms.txt 和变更清单
```

机器接口、人类页面和搜索索引必须来自同一份审核数据，避免不同入口给出互相矛盾的信息。

### 11.2 节省服务器资源

- 大部分目录页和详情页静态生成；
- JSON/JSONL 使用预生成快照和压缩传输；
- 搜索优先使用小型本地索引，不在每次请求时扫描所有 YAML；
- API 使用缓存、分页和速率限制；
- 不保存 GitHub 仓库代码；
- GitHub 元数据按计划增量更新，不持续全量爬取；
- 图谱可视化按需加载，不作为首页必需资源；
- 中转站和目录网站暂时可同机，但进程、端口、数据和日志必须隔离，并设置资源上限。

### 11.3 数据更新

每条项目记录区分：

- `source_updated_at`：上游 GitHub 最近更新时间；
- `verified_at`：本站最近核实时间；
- `verification_status`：已核实、待复核、失效、归档；
- `evidence_urls`：字段依据；
- `content_hash`：上次核实时的关键内容摘要。

更新策略：

1. 定时读取仓库是否存在、默认分支、许可证和最近更新时间；
2. 关键字段变化时进入待复核队列，不自动覆盖编辑结论；
3. 智能体在自主访问上游仓库时发现失效、改名、版本变化或描述偏差，可提交结构化差异反馈；没有发现变化时不产生报告；
4. 外部反馈先进入隔离区，经规则与智能体审核并附可追溯证据后才能进入正式目录；仅在自动化阻塞时人工介入；
5. 删除或私有化的仓库不立即抹除历史记录，而标记状态并保留最后核实证据。

### 11.4 路过式差异反馈

差异反馈不是任务分发系统。本站不要求智能体领取一批仓库，也不要求每次访问都交作业。它只利用智能体原本就会发生的行为：

```text
智能体因自身需求在本站找到项目
→ 点击或读取上游 GitHub 仓库
→ 顺手比较本站记录与上游现实
→ 没有明显差异：不提交任何内容
→ 发现明显差异：提交一条最小差异报告
```

允许上报的变化类型：

```text
repository_missing          # 仓库不存在或被删除
repository_private          # 仓库已转为私有
repository_redirected       # 仓库改名或地址跳转
repository_archived         # 项目被归档
repository_reactivated      # 项目重新活跃
release_changed             # 出现重要新版本
license_changed             # 许可证变化
summary_mismatch            # 本站简介与上游用途明显不符
deployment_changed          # 安装或部署方式变化
maintenance_changed         # 维护状态变化
other_material_change       # 其他实质变化
```

第一阶段的报告边界非常窄：智能体只比较“本站已经展示的地址、简介、版本、许可证、归档和维护状态”与它实际打开上游后看到的事实。它不领取检查任务，不提交泛化研究，不重写完整项目介绍，也不顺便推荐新项目。新项目推荐、安全研究和一般贡献使用其他贡献入口，不进入这条路过式差异反馈通道。

最小差异报告：

```json
{
  "project_id": "example-project",
  "baseline_hash": "本站当前记录哈希",
  "change_type": "repository_missing",
  "observed_value": "GitHub 返回 404",
  "short_description": "原仓库地址已经无法访问",
  "evidence_url": "https://github.com/example/project",
  "upstream_fingerprint": "404",
  "observed_at": "2026-07-17T12:00:00Z"
}
```

报告只回答：本站原记录与上游现实哪里不对应、现在实际是什么、证据在哪里。智能体只写几句话，不需要解释检查过程，也不需要重写完整项目介绍。

反馈收件箱必须按事件指纹实时去重：

```text
project_id
+ baseline_hash
+ change_type
+ upstream_fingerprint
= event_fingerprint
```

第一个报告创建变化事件；后来发现相同变化的智能体只增加确认次数，不再保存一份重复报告。确认次数达到可信上限后，后续相同报告直接丢弃。

每天固定时间只生成一份去重后的变化清单，按照紧急、重要、普通分组。Codex 逐项打开证据，重新核实后才能修改正式数据。外部报告永远不能自动覆盖项目记录。

处理完成后：

- 删除临时原始报告、重复报告和收件箱事件；
- 保留正式数据修改、Git 提交、修改理由和证据链接；
- 当日没有实质变化时，不生成空报告；
- 不为“没有变化”的普通访问保存逐次日志。

这样即使访问量非常大，系统需要处理的数量也由“真实发生了多少种变化”决定，而不是由“来了多少智能体”决定。

## 12. 分阶段上线门槛

### 阶段 0：本地规格与数据整顿

- 完成新字段设计；
- 将现有 389 个项目转换到新结构；
- 建立校验、快照和检索评测；
- 建立广告字段，但不接真实广告；
- 不注册广告账户；
- 不公开部署。

### 阶段 1：本地可用原型

- JSON/JSONL；
- 本地搜索 API；
- MCP；
- 简单人类页面；
- 路过式差异反馈的数据格式、去重规则和每日变化清单生成器；
- 机器和人类入口中的赞助示例均明确标记；
- 自动化测试确认付费内容不会进入客观排名。

### 阶段 2：无广告公开测试

前置条件：

- 用户明确批准公开；
- 域名、HTTPS 和部署方式确定；
- 隐私、Cookie、数据来源、广告政策和联系页面完成；
- 日志最小化和限速完成；
- 差异反馈收件箱能够去重、限流，且不能直接修改正式数据；
- 服务器资源监控完成；
- 中转站与目录网站相互隔离；
- 明确是否需要 ICP、经营许可或其他面向目标市场的手续。

先验证：

- 真人是否真的使用分类和比较；
- 智能体是否读取机器接口；
- GitHub 跳转是否增长；
- 服务器成本是否可控；
- 数据更新流程是否能持续。

### 阶段 3：真人广告测试

前置条件：

- 确定个人、个体工商户或公司主体；
- 确定同名收款账户；
- 向银行确认收款材料；
- 向 12366 或税务师确认税目、发票和跨境服务处理；
- 选择正确的广告账户类型；
- 欧洲相关同意机制就绪；
- 页面具备足够原创编辑价值。

### 阶段 4：透明的智能体赞助测试

前置条件：

- 有真实广告主和书面合同；
- 有广告主身份审核；
- 有机器可读的 Sponsored 标记；
- 有同内容的人类可见页面；
- 有独立于客观排名的展示位；
- 有展示、下架、结算和投诉记录；
- 税务和外汇处理已经得到确认；
- 不采用隐藏、机器人专属或不披露的注入方式。

## 13. 成功标准

第一版成功不以广告收入衡量，而以以下结果衡量：

1. 智能体可用结构化条件找到合适的 1—3 个候选；
2. 每个推荐能解释适用与不适用场景；
3. 机器入口和人类页面结果一致；
4. 项目来源、核实时间和维护状态清晰；
5. 现有 389 个项目可转换且不丢失原信息；
6. 赞助内容无法影响客观排名；
7. 智能体发现差异时可提交最小报告，相同事件不会形成重复报告；
8. 每天只生成一份去重后的变化清单，并且外部反馈不能自动修改正式数据；
9. 日志和服务器成本可控；
10. 尚未具备收款条件时不会提前形成难以解释的跨境收入；
11. 一旦开始收款，可形成合同、服务、资金和税务的完整证据链。

## 14. 当前明确不做

- 不全量抓取 GitHub 一亿级仓库；
- 不建立“领取一批仓库才能使用目录”的任务系统；
- 不要求每个智能体或每次无变化访问都提交报告；
- 不承诺智能体爬取一定产生广告收入；
- 不用机器人刷 Google 或其他真人广告；
- 不做隐藏赞助、搜索引擎 cloaking 或不同用户看到互相矛盾的商业内容；
- 不立即注册公司、购买域名或开通广告账户；
- 不立即公开网站；
- 不迁移现有中转站；
- 不把身份证、银行、合同和税务材料传到公开 GitHub；
- 不在第一版开发账号、评论、社区和复杂可视化。

## 15. 正式运营前必须人工确认的清单

在出现第一笔真实广告收入前，用户应携带业务说明和合同样本，分别确认：

### 向广告平台确认

- 中国主体是否可注册；
- 接受个人、个体工商户还是公司；
- 合同相对方是谁；
- 支持的收款币种和银行方式；
- 可提供哪些结算和税务材料；
- 账户类型以后能否变更；
- 智能体流量的有效展示定义；
- 无效流量和退款规则。

### 向银行确认

- 推荐使用个人账户、个体户账户还是公司账户；
- 境外广告服务收入的入账用途；
- 每笔或每月需要的合同、结算单、发票和付款凭证；
- 是否支持所选广告平台的付款路径；
- 结汇和国际收支申报如何填写。

### 向 12366、主管税务机关或税务师确认

- 所得来源与税目；
- 自然人、个体户或公司哪种处理与实际业务相符；
- 是否需要开具数电发票或使用其他出口服务凭证；
- 增值税、附加税、个人所得税或企业所得税；
- 是否符合跨境交易优惠及需要的备案资料；
- 广告档案、会计档案和税务资料的保存要求。

确认结果应形成书面备忘并归入敏感资料目录。

## 17. 设计结论

本项目可以继续做，但必须按以下顺序推进：

```text
先把开源项目目录做出真实价值
→ 再验证真人和智能体访问
→ 接收智能体路过式差异反馈并每日核实
→ 再确定中国收款主体
→ 再开真人广告
→ 最后小规模测试透明的智能体赞助
```

广告是目录价值形成后的变现层，不是用机器人空爬就能自动赚钱的捷径。项目的核心资产仍然是：可核实的数据、好用的分类、可靠的取舍、稳定的机器接口和长期更新能力。
